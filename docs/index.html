院内問題ボードに、記憶を持たせる

さくら歯科クリニックの院内問題ボードを、データベース（Neon）と繋ぎ、誰の画面からも同じ記録が見えるようにした

公開したアプリを開く（dental-board-five.vercel.app）
導入前
リロードすると入力がすべて消える。自分のパソコンでしか見られない
導入後
リロードしても残る。別の端末やスマホからも同じ内容が見える

このツールが解決する課題

3ヶ月目に設計した内容 ── 院内の問題が「誰も知らないまま」放置される状況を変える

スタッフ

院内問題ボード
集約・可視化
院長・副院長
状況を把握
スタッフ
問題に気づいても、どこに・どうやって報告すればいいか明確でなかった

導入前の課題
  • 問題が個人メモ・口頭・LINEに分散して放置
  • 誰が何の問題を把握しているか見えない

  • 問題をボードに集約し、全件を一か所で把握
  • 遠隔の副院長もリアルタイムで状況確認できる

保持できるようにしたデータ

1つの「問題」に対して、コメントが何件でもつながる形で保存している

例：実際にはこんな形で保存されている
問題：滅菌パックの封が甘い
機器・設備／緊急／報告者：伊藤さん／担当者：佐藤さん
コメント：伊藤さん（スタッフ）
今朝、封が甘いパックが2つ見つかった
コメント：佐藤さん（担当者）
滅菌機を点検します

このように「問題」1件に対して、コメントが0件・1件・2件…と自由につながる。逆にコメント側は必ず「自分がどの問題への返信か」を持っている。

もし1つにまとめて保存したら
コメントを1件増やすたびに、同じ問題の情報（タイトル・カテゴリ・担当者など）を毎回書き写すことになり、同じ内容が何度も重複してしまう
問題とコメントを分けたので
問題の情報は1回だけ保存すればよく、コメントは「どの問題への返信か」という印だけ持てばよい。重複がなく、あとから見返しやすい

  • 分類・優先度
  • 対応状況・担当者
  • 報告者・その役職
  • 対応期限・登録日

  • 投稿日時

保存先と、それを選んだ理由

Vercel経由で、データベースサービス「Neon」を選択

Markdown
自分ひとりのメモなら十分。複数人での同時利用には不向き
JSON
機械的に扱いやすいが、複数人が同時に書き換えると壊れやすい
データベース（採用）
スタッフ・院長・副院長が同時に読み書きしても矛盾なく扱える
選んだ理由：関係者が同時にアクセスするツールだったこと
  • スタッフ・院長・遠隔の副院長が同時に同じ画面を見る前提のツールのため、1人のブラウザにしか残らないlocalStorageでは成立しない
  • 問題とコメントのように「1つに対して複数」つながるデータは、表を分けてひもづけられるデータベースの方が扱いやすい
  • データベースにつなぐための鍵（接続情報）は、コードとは別の「Vercelの設定画面」だけに保管した。コードは誰でも見られる場所に置いているが、データベースの鍵は関係者以外に見せたくないため、あえて保管場所を分けている
表の分け方も、医院の規模に合わせて選んだ
1つの表にまとめる
コメントが増えるたびに同じ行を書き換える必要があり、同時利用に弱い
問題とコメントだけ分ける（採用）
「1つに対して複数」つながる関係だけを表に分け、スタッフやカテゴリは固定リストのまま扱う
スタッフ・カテゴリも表に分ける
大人数の医院やスタッフの入れ替えが多い場合向き。今回の規模には過剰

1医院・スタッフ数名〜十数名という個人経営規模では、スタッフやカテゴリまで表に分けても管理の手間が増えるだけで恩恵が薄い。「問題とコメントが1つに対して複数つながる」という、今後も変わらない関係だけを表に分け、それ以外は今の身の丈に合わせてシンプルに保つという判断をした。

工夫したポイントと苦戦したポイント

ローカル動作の確認から公開まで、実際につまずいた箇所

工夫したポイント
  • 1
    ログイン無しの「報告者選択」
    本格的なログイン機能は作らず、投稿時に自分の名前をプルダウンで選ぶだけの軽い仕組みにした。一度選ぶとブラウザに記憶され、次回から自動選択される
  • 2
    設定情報が無くても公開作業が止まらない設計
    データベースへの接続を、実際に使う瞬間まで遅らせる作りにした。これにより、Vercelでアプリを公開する作業の途中で、接続情報がまだ無くてもエラーにならないようにした
  • 3
    画面のコードはほぼ変えない
    3ヶ月目に作った画面の見た目・操作感はそのままに、データの出し入れの仕組みだけをデータベースとやり取りする形に差し替えた
  • 4
    「繋がった」という報告を鵜呑みにしない
    AIに作業を任せると「データベースに繋がりました」と言われても、実は繋がっていないことがある。今回は公開後、実際に公開URLへ直接アクセスして本物のデータが返ってくることを目で確認してから完了とした
苦戦したポイント
  • データベースから返ってくる結果の形が、想定と違った
    問い合わせの方法によって、結果の形（データの並び方）に微妙な違いがあり、アプリを公開する作業のたびに3回連続でエラーになった。1つずつ実際の結果を確認しながら直した
  • 公開の直前で、公開作業が止められた
    コードを保存した際に使ったメールアドレスと、GitHubアカウントのメールアドレスが一致せず、Vercelから安全のため公開を止められた。アカウントの設定を合わせることで解決した
  • 設定情報の名前のズレに、気づきにくい
    データベースと連携する際、保存する情報の名前を自由に決められる仕組みになっており、コード側が読み込む名前とズレると、エラーすら出ないまま「つながらない」状態になりかねなかった。作成時に名前を指定してそろえることで解決した
