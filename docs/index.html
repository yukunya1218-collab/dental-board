院内運営ツール ／ データベース連携

院内問題ボードに、記憶を持たせる

ブラウザを閉じると消えていたデータを、Neon（PostgreSQL）に保存し、誰の画面からも同じ記録が見えるようにした

ひとことで言えば：画面の中だけの記録を、データベースの記録に変えた

導入前
リロードすると入力がすべて消える。自分のパソコンでしか見られない
導入後
リロードしても残る。別の端末やスマホからも同じ内容が見える

4つのペイン（絞り込み・一覧・詳細・対応管理）の内容は、すべてデータベースから読み込まれている

保持できるようにしたデータ

「問題」と「コメント」の2つの表に分け、1対多で紐づけた

problems（問題）
  • id主キー
  • title / descriptionタイトル・詳細
  • category / priority分類・優先度
  • status / assignee対応状況・担当者
  • reported_by報告者・役職
  • deadline / created_at期限・登録日
1つの問題に
複数のコメント
comments（コメント）
  • id主キー
  • problem_idproblems.id への参照
  • author / role投稿者・役職
  • textコメント本文
  • created_at投稿日時

問題1件に対してコメントは0件以上つく関係のため、別の表に分けて problem_id で紐づけた。1つの表にすべて詰め込むと、コメントの数だけ問題の情報が重複してしまうため。

保存先と、それを選んだ理由

Vercel経由でNeon（サーバーレスPostgres）を選択

Markdown
自分ひとりのメモなら十分。複数人での同時利用には不向き
JSON
機械的に扱いやすいが、複数人が同時に書き換えると壊れやすい
データベース（採用）
スタッフ・院長・副院長が同時に読み書きしても矛盾なく扱える
選んだ理由：関係者が同時にアクセスするツールだったこと
  • スタッフ・院長・遠隔の副院長が同時に同じ画面を見る前提のツールのため、1人のブラウザにしか残らないlocalStorageでは成立しない
  • 問題とコメントのように「1対多」の関係を持つデータは、表を分けて紐づけられるデータベースの方が扱いやすい
  • 接続情報（DATABASE_URL）はVercelの環境変数として保管し、コードのリポジトリには含めないようにした。AIが自由に読めるリポジトリと、鍵がないと開けないデータベースを意図的に分けている

工夫したポイントと苦戦したポイント

ローカル動作の確認から公開まで、実際につまずいた箇所

工夫したポイント
  • 1
    ログイン無しの「報告者選択」
    本格的なログイン機能は作らず、投稿時に自分の名前をプルダウンで選ぶだけの軽い仕組みにした。一度選ぶとブラウザに記憶され、次回から自動選択される
  • 2
    環境変数が無くてもビルドが壊れない設計
    データベース接続を、実際に使う瞬間まで遅らせる作りにした。これにより、Vercelのビルド時に接続情報が無くてもエラーにならないようにした
  • 3
    画面のコードはほぼ変えない
    前回作った画面の見た目・操作感はそのままに、データの出し入れ部分だけをAPI経由に差し替えた
苦戦したポイント
  • データベースからの戻り値の形が想定と違った
    問い合わせ方法によって、結果が配列そのものだったり配列を包んだ形だったりと差があり、ビルドが3回連続で型エラーになった。1つずつ実際の戻り値を確認して修正した
  • 公開直前にデプロイがブロックされた
    コードを保存した際のメールアドレスと、GitHubアカウントのメールアドレスが一致せず、Vercelに公開を止められた。設定を実際のアカウントに合わせて解決した
  • 環境変数の名前ズレに気づきにくい
    データベース連携時に変数名を自由に変えられる設定があり、コードが読み込む名前とズレるとエラーにすら出ずに接続できない状態になりかねなかった。作成時に名前を指定して統一した
